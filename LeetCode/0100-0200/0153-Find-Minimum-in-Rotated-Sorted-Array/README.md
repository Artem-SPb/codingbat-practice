# [153. Find Minimum in Rotated Sorted Array](https://leetcode.com/problems/find-minimum-in-rotated-sorted-array/)

## Описание задачи

Представьте массив уникальных чисел, отсортированный по возрастанию. Затем этот массив "повернули" (циклически сдвинули) от `1` до `n` раз.
Например, массив `[0,1,2,4,5,6,7]` после 4 сдвигов может стать `[4,5,6,7,0,1,2]`.

Вам дан такой повернутый массив `nums`. Необходимо найти минимальный элемент в нем.

**Строгое ограничение:** Алгоритм должен работать за время **O(log n)**.

**Пример 1:**

- **Input:** nums = [3,4,5,1,2]
- **Output:** 1
- **Пояснение:** Оригинальный массив был [1,2,3,4,5], его повернули 3 раза. Минимум — 1.

**Пример 2:**

- **Input:** nums = [4,5,6,7,0,1,2]
- **Output:** 0

**Пример 3:**

- **Input:** nums = [11,13,15,17]
- **Output:** 11
- **Пояснение:** Массив повернули 4 раза (вернув в исходное состояние). Минимум — 11.

---

## Мой подход к решению

### 1. Размышления (Thought Process)

- **Анализ структуры:** Если массив был отсортирован, а затем повернут, он теперь состоит из **двух** возрастающих отрезков. Причем все числа в левом отрезке строго больше всех чисел в правом отрезке.
- **Где минимум?** Минимум — это всегда самая первая точка второго отрезка (место "обрыва").
- **Бинарный поиск аномалии:** Нам не нужно искать конкретное число. Нам нужно понять, в какой половине находится "обрыв". Для этого мы будем сравнивать элемент посередине (`nums[mid]`) с самым правым элементом (`nums[right]`).
  1. Если `nums[mid] > nums[right]` (например, середина `7`, а край `2`), это значит, что мы сейчас стоим на высокой скале левого отрезка. "Обрыв" и минимум точно находятся **справа**. Мы сдвигаем `left = mid + 1`.
  2. Если `nums[mid] <= nums[right]` (например, середина `1`, а край `2`), это значит, что мы уже спустились на правый (нижний) отрезок. Минимум либо прямо под нами, либо **слева**. Мы сдвигаем `right = mid`. Мы не можем сделать `mid - 1`, потому что сам `mid` может оказаться минимумом!

### 2. Алгоритм

1. Инициализировать указатели: `left = 0`, `right = len(nums) - 1`.
2. Пока `left < right` (обратите внимание, без знака `=`, так как мы ищем схождение указателей в одной точке):
   - Найти середину `mid = (left + right) // 2`.
   - Если `nums[mid] > nums[right]`: сдвинуть левую границу (`left = mid + 1`).
   - Иначе: сдвинуть правую границу (`right = mid`).
3. Когда цикл прервется, `left` и `right` встретятся на минимальном элементе. Вернуть `nums[left]`.

---

## Анализ сложности

- **Сложность по времени (Time Complexity):** **O(log N)**. На каждом шаге мы отбрасываем ровно половину массива, что является классическим логарифмическим временем.
- **Сложность по памяти (Space Complexity):** **O(1)**. Мы используем только три переменные (`left`, `right`, `mid`), дополнительная память не требуется.
