# [33. Search in Rotated Sorted Array](https://leetcode.com/problems/search-in-rotated-sorted-array/)

## Описание задачи

Массив целых чисел `nums` был изначально отсортирован по возрастанию (содержит только уникальные значения). Затем он был повернут (сдвинут) вокруг неизвестного индекса `k`. 
Например, массив `[0,1,2,4,5,6,7]` после сдвига на 4 позиции может превратиться в `[4,5,6,7,0,1,2]`.

Вам дан этот массив `nums` и целое число `target`. Необходимо найти `target` в массиве и вернуть его индекс. Если число в массиве отсутствует — вернуть `-1`.

**Ограничение:** Алгоритм должен иметь временную сложность строго `O(log n)`.

**Пример 1:**
* **Input:** nums = [4,5,6,7,0,1,2], target = 0
* **Output:** 4

**Пример 2:**
* **Input:** nums = [4,5,6,7,0,1,2], target = 3
* **Output:** -1

**Пример 3:**
* **Input:** nums = [1], target = 0
* **Output:** -1

---

## Мой подход к решению

### 1. Размышления (Thought Process)
* **Жёсткое ограничение:** Требование `O(log n)` мгновенно отсекает любой линейный проход по массиву. Единственный алгоритм поиска с такой сложностью — это **Бинарный поиск**.
* **В чём проблема:** Классический бинарный поиск требует полностью отсортированного массива. Наш массив состоит из двух отсортированных частей (например: `[4,5,6,7]` и `[0,1,2]`).
* **Ключевой инсайт:** Если мы возьмем середину массива `mid`, то **хотя бы одна из половинок (левая или правая) гарантированно будет строго отсортирована**.
  * Как проверить, отсортирована ли левая часть `[left ... mid]`? Если `nums[left] <= nums[mid]`, значит на этом отрезке разрыва нет.
  * Если левая часть отсортирована, мы легко можем проверить, находится ли наш `target` внутри диапазона `nums[left] <= target < nums[mid]`. Если да — сужаем поиск в левую половину (`right = mid - 1`). Если нет — ищем в правой.
  * Зеркальная логика применяется, если отсортированной оказалась правая половина.

### 2. Алгоритм
1. Инициализируем классические указатели бинарного поиска: `left = 0`, `right = len(nums) - 1`.
2. Запускаем цикл пока `left <= right`:
   * Вычисляем серединный индекс: `mid = (left + right) // 2`.
   * Если `nums[mid] == target`, мы нашли ответ — возвращаем `mid`.
   * **Проверяем левую половину:** Если `nums[left] <= nums[mid]` (левая часть отсортирована):
     * Если `nums[left] <= target < nums[mid]`, сдвигаем правый указатель: `right = mid - 1`.
     * Иначе искомое число в правой части: `left = mid + 1`.
   * **Иначе (отсортирована правая половина):**
     * Если `nums[mid] < target <= nums[right]`, ищем в правой части: `left = mid + 1`.
     * Иначе сдвигаем левый указатель: `right = mid - 1`.
3. Если цикл завершился и число не найдено, возвращаем `-1`.

---

## Анализ сложности

* **Сложность по времени (Time Complexity):** **O(log n)**. На каждом шаге мы отсекаем ровно половину массива, сохраняя логарифмическую скорость классического бинарного поиска.
* **Сложность по памяти (Space Complexity):** **O(1)**. Мы используем только три переменные для индексов (`left`, `right`, `mid`), дополнительная память не выделяется.
