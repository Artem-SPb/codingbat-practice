# [42. Trapping Rain Water](https://leetcode.com/problems/trapping-rain-water/)

## Описание задачи

Дан массив неотрицательных целых чисел `height`, представляющий карту высот, где ширина каждого столбика равна 1. Необходимо вычислить, сколько воды он сможет удержать после дождя.

**Пример:**
**Input:** height = [0,1,0,2,1,0,1,3,2,1,2,1]
**Output:** 6
**Пояснение:** Карта высот представлена массивом. В углублениях между столбиками скапливается 6 единиц воды.

---

## Мой подход к решению

### 1. Размышления (Thought Process)

- **Базовая идея:** Количество воды над любым столбиком `i` зависит от максимальной высоты столбика слева от него (`max_left`) и максимальной высоты справа (`max_right`). Вода, которая скопится над текущим столбиком, равна `min(max_left, max_right) - height[i]`.
- **Подход с дополнительной памятью:** Можно создать два массива, чтобы заранее вычислить `max_left` и `max_right` для каждой позиции. Это даст время O(n), но потребует O(n) дополнительной памяти.
- **Оптимальный подход (Two Pointers):** Мы можем использовать два указателя — `left` (идет от начала) и `right` (идет с конца). Нам не нужно знать _абсолютный_ максимум с другой стороны. Если `max_left < max_right`, то уровень воды жестко ограничен именно левой стороной (и наоборот). Это позволяет вычислять объем воды "на лету", сократив затраты памяти до O(1).

### 2. Алгоритм

1. Инициализировать указатели `left = 0` и `right = len(height) - 1`.
2. Завести переменные для отслеживания максимумов: `max_left = height[left]` и `max_right = height[right]`.
3. Переменная `water = 0` будет хранить итоговый результат.
4. Пока `left < right`:
   - Сравниваем `max_left` и `max_right`.
   - Если `max_left < max_right`: сдвигаем `left` вправо. Обновляем `max_left`. Если текущая высота меньше `max_left`, добавляем разницу в `water`.
   - Иначе: сдвигаем `right` влево. Обновляем `max_right`. Если текущая высота меньше `max_right`, добавляем разницу в `water`.
5. Вернуть `water`.

---

## Анализ сложности

- **Сложность по времени (Time Complexity):** **O(n)**, так как мы проходим по массиву ровно один раз сближающимися указателями.
- **Сложность по памяти (Space Complexity):** **O(1)**, используется только несколько переменных для указателей и аккумулятора ответа.

---

## Ключевые выводы / Паттерн

- **Two Pointers (Два указателя):** Классический пример оптимизации памяти. Вместо прекалькуляции массивов максимумов, мы динамически принимаем решение о сдвиге того указателя, чей локальный максимум на данный момент меньше.
